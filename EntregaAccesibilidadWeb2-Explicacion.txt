La web a analizar es la siguiente:http://www.resinformatica.com.ar/LOGIN_ONLY/login-only/GlobalBluePoint-ERP.aspx
Analizaremos concretamente la pestaña de login del sistema para demostrar que podriamos
hacer una pestaña de formularios que sea accesible.

En este caso los cambios realizados fueron los siguientes:
(Estos pueden visualizarse en el archivo HTML también, realizando una busqueda
que contenga el string 'EDITADO' para visualizar algunos de estos casos con mayor facilidad)

1°) En varios casos hicimos uso del atributo alt para escribir un texto alternativo 
para las imágenes, con el fin de que pueda entenderse el contexto de la página 
con la misma facilidad para aquellos usuarios que no puedan visualizarlas.
2°) Se añadieron descripciones de links de redirecciones (originalmente no las contenian).
3°) Se agrandaron varias imágenes para que cumplan con el tamaño minimo recomendado por la wcag
(16 px).
4°) Corrección de textos alternativos que no describian claramente la operación que realizaba
un determinado componente del html (Por ejemplo, se hace referencia a que en lugar de contener el texto 'Ir', se cambió
por 'Dirigirse a la página x').
5°) Se añadieron atributos 'keypress', para que puedan accederse a los elementos no solo usando el mouse, sino
también utilizando el teclado.
6°) Se aclaró el lenguaje de la pagina en el encabezado del html.


En este caso al validar la página después con la herramienta 'Simor',
los errores que surgieron son:
-Regla 83 de la wcag: Los formularios deben tener id's que no se repitan en ningun otro formulario.
(Esto no altera la forma en que se percibe el sistema para el usuario, pero ayudaría a reducir la probabilidad
de errores en el desarrollo)
-Regla 44: Los links deben ser descriptivos. 
Por ejemplo, como mencionamos, el link analizado es:
    www.resinformatica.com.ar/LOGIN_ONLY/login-only/GlobalBluePoint-ERP.aspx
este link muestra el login del sistema, por lo tanto sería preferible que el link del
mismo fuese por ejemplo: www.resinformatica.com.ar/Iniciar_sesion
